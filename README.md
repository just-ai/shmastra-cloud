# Project Specification: Shmastra Cloud

## Overview

A Next.js 15 application deployed on Vercel that hosts Mastra Studio as a
protected SPA for authenticated users. Each user gets their own isolated Mastra
server running inside an E2B sandbox. The Next.js app acts as the central proxy
layer: it forwards Studio UI requests to the user's sandbox and forwards all AI
provider API calls (OpenAI, Anthropic, Gemini, Composio) using a virtual key
system that keeps real API keys out of the sandbox entirely.

---

## Tech Stack

- **Framework**: Next.js 15 (App Router), TypeScript, deployed on Vercel
- **Auth**: WorkOS AuthKit (`@workos-inc/authkit-nextjs`, `@workos-inc/node`) —
  Magic Auth (email OTP)
- **Sandbox**: E2B (`e2b` latest) — one sandbox per user, hibernates on inactivity
- **Database**: Supabase (Postgres) — stores user profiles, sandbox metadata,
  virtual keys
- **API routing**: Hono (`hono`) with `hono/vercel` adapter for the proxy routes
- **Studio build**: Vite (`vite`) — copies Mastra Studio static assets from
  `node_modules/mastra/dist/studio` into `public/studio/` at build time
- **Mastra**: `mastra` (latest) — provides the Studio SPA assets. The Mastra
  server that runs inside E2B is cloned from
  [github.com/just-ai/shmastra](https://github.com/just-ai/shmastra)

## Key Design Decisions

### 1. Studio SPA served from Next.js `public/`

Mastra Studio is a React SPA whose static assets live inside
`node_modules/mastra/dist/studio`. A Vite build step (run as part of
`npm run build`) copies those assets into `public/studio/` and replaces
`%%PLACEHOLDER%%` strings in `index.html` with values from `.env`.

Because `MASTRA_SERVER_HOST` is left empty, Studio constructs all API URLs
relative to its own origin. This means every Studio request goes to the
same domain as the Next.js app — no hardcoded Mastra server URL baked into
the build. The Next.js rewrite rule sends `/studio` and `/studio/*`
(HTML requests only) to `public/studio/index.html`; static JS/CSS assets are
served directly from `public/studio/` by Next.js.

### 2. One sandbox per user, hibernation on inactivity

Each user gets exactly one E2B sandbox created at first login. The
sandbox runs a Mastra dev server on port 4111. After 10 minutes of inactivity
the sandbox automatically pauses itself (`lifecycle: { onTimeout: 'pause' }`).
When Studio makes its next request, `Sandbox.connect(sandboxId)` resumes it
in ~1 second without needing to recreate it.

The sandbox ID is stored in the Supabase `sandboxes` table.

### 3. Async sandbox creation with webhook + polling

Sandbox creation can take 30–60 seconds and cannot block the HTTP response
(Vercel would time out). The flow is:

1. After first successful login, the app checks if the user has a sandbox.
   If not, it writes `status = 'creating'` to the `sandboxes` table, calls
   `after(createSandboxForUser(userId))` (Next.js 15 `after()`) and shows
   a "Setting up your environment..." screen.
2. Frontend polls `GET /api/sandbox/status` every 2 seconds.
3. Inside the sandbox, the startup script starts the Mastra dev server, waits
   for `http://localhost:4111` to respond, then sends
   `POST /api/sandbox/ready` with `{ sandboxId, token, userId }`.
4. `/api/sandbox/ready` verifies the one-time token from the `sandboxes`
   table, sets `status = 'ready'`, and deletes the token.
5. Frontend receives `{ status: 'ready' }` and redirects to `/studio`.

The one-time `READY_TOKEN` is a 16-byte random hex string generated before
`Sandbox.create()`, stored in the `sandboxes` table, and passed to the sandbox
via `envs`. E2B automatically injects `E2B_SANDBOX_ID` inside the sandbox so
the startup script can reference it.

### 4. Virtual key system for AI providers

Real API keys (OpenAI, Anthropic, Gemini, Composio) are stored in Vercel
environment variables and are **never** sent to the sandbox. Instead:

- At sandbox creation time, `getOrCreateVirtualKey(userId)` generates a
  single virtual key in the format `vk_{userId}_{24hex}` and stores it in
  the Supabase `virtual_keys` table.
- The sandbox receives this virtual key as all four provider env vars:
  `OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, `GEMINI_API_KEY`, `COMPOSIO_API_KEY`.
- The sandbox also receives overridden base URLs pointing back to Next.js:
  `OPENAI_BASE_URL=https://app/api/llm/openai`,
  `ANTHROPIC_BASE_URL=https://app/api/llm/anthropic`,
  `GEMINI_BASE_URL=https://app/api/llm/gemini`,
  `COMPOSIO_BASE_URL=https://app/api/llm/composio`.
- When the Mastra project inside the sandbox calls OpenAI/Anthropic/etc.,
  the SDK sends the virtual key to `https://app/api/llm/{provider}/...`.
- The Hono `/api/llm/:provider/*` handler extracts the virtual key from the
  `Authorization` or `x-api-key` header, calls `resolveVirtualKey(vk)` which
  looks up the user in the Supabase `virtual_keys` table, verifies the key
  is valid, retrieves the real API key from `process.env`, swaps the header,
  and proxies the request to the real provider.

This per-user virtual key design allows future rate limiting or billing per
user without changing the architecture.

### 5. Hono for proxy routes — SSE/streaming safe

Both proxy routes (`/api/mastra/*` and `/api/llm/:provider/*`) use Hono with
`hono/vercel` adapter in a single catch-all route
`app/api/[[...route]]/route.ts`. Hono's `stream(c, s => s.pipe(upstream.body))`
pipes the upstream `ReadableStream` directly to the client with zero
buffering. The `X-Accel-Buffering: no` header disables Vercel/nginx
buffering. This is critical for SSE streams from OpenAI and Anthropic
streaming responses, which Studio uses for live agent output.

### 6. Auth: WorkOS Magic Auth (Email OTP)

Authentication uses WorkOS AuthKit with **Magic Auth** (email one-time code).
No passwords.

- The login page is a custom modern UI with an email input field. After
  submitting the email, the user enters the OTP code received via email.
- WorkOS AuthKit handles the OTP sending, verification, and session
  management via `@workos-inc/authkit-nextjs`.

### 7. Supabase as the data store

All application state is stored in Supabase (Postgres). WorkOS is used
exclusively for authentication (OTP + session). No data is stored in WorkOS
`user.metadata`.

### 8. AI provider keys are shared

All users share the same AI provider keys (OpenAI, Anthropic, Gemini,
Composio) stored in Vercel environment variables. The virtual key system
exists to:
- Authenticate requests from sandboxes (verify the request comes from a
  known user)
- Enable future per-user rate limiting or billing

## Build Scripts

### `npm run build:studio`
Runs `vite build --config studio-build/vite.config.js`. Copies
`node_modules/mastra/dist/studio` to `public/studio/`, then replaces
`%%PLACEHOLDER%%` patterns in `public/studio/index.html` with values from
`.env.local` (falling back to `.env`) for `MASTRA_*` variables. Output
directory is `public/studio/` (should be
gitignored).

### `npm run build`
Runs `build:studio` then `next build`. This is the command Vercel uses.

### `npm run template:build`
Runs `tsx scripts/build-e2b-template.ts`. Uses E2B SDK to build a sandbox
template:
```ts
// Pseudocode for build-e2b-template.ts
Template()
  .fromImage('node:22-slim')
  .aptInstall(['curl', 'git'])
  .makeDir('/app', { user: 'root' })
  .runCmd('chown user:user /app', { user: 'root' })
  .runCmd('git clone https://github.com/just-ai/shmastra /app')
  .runCmd('cd /app && npx pnpm install')
  .runCmd('cd /app && npx -y playwright@latest install chromium')
  .copy('./start.sh', '/app/start.sh')
  .runCmd('chmod +x /app/start.sh')
  .setWorkdir('/app')
```
Built with alias `'mastra-server'`. Must be re-run whenever the shmastra
repo changes. Requires `E2B_API_KEY` in env.

## Mastra Project Inside Sandbox

The Mastra project is cloned from
[github.com/just-ai/shmastra](https://github.com/just-ai/shmastra) during the
E2B template build. It is **not** part of this repository.

The sandbox receives overridden base URLs and virtual keys via environment
variables at sandbox creation time:
```
OPENAI_API_KEY=vk_...          OPENAI_BASE_URL=https://app.vercel.app/api/llm/openai
ANTHROPIC_API_KEY=vk_...       ANTHROPIC_BASE_URL=https://app.vercel.app/api/llm/anthropic
GEMINI_API_KEY=vk_...          GEMINI_BASE_URL=https://app.vercel.app/api/llm/gemini
COMPOSIO_API_KEY=vk_...        COMPOSIO_BASE_URL=https://app.vercel.app/api/llm/composio
```

The template does not auto-start the Mastra server. It only prepares `/app`
and includes `/app/start.sh` as a convenience script for manual startup inside
the sandbox.

- Email input field
- OTP code input (shown after email submission)
- WorkOS AuthKit Magic Auth integration
- Redirect to `/studio` after successful auth (or to sandbox setup screen

### 2. Root redirect (`app/page.tsx`)
Redirect authenticated users to `/studio`, unauthenticated to `/login`.

### 3. WorkOS AuthKit integration
- Configure Magic Auth in the WorkOS Dashboard.
- Set up `@workos-inc/authkit-nextjs`: callback route, middleware,
  `AuthKitProvider`, `WORKOS_COOKIE_PASSWORD`.

### 4. Supabase setup
- Create Supabase project.
- Run `supabase/migrations/001_init.sql` in the Supabase SQL Editor.
- Add `SUPABASE_URL` and `SUPABASE_SERVICE_ROLE_KEY` to Vercel env and
  `.env.local`.

### 5. E2B template build script (`scripts/build-e2b-template.ts`)
- Implement the template builder that clones `just-ai/shmastra`, installs
  deps with pnpm, installs Playwright Chromium, and includes `start.sh` for
  manual startup inside the sandbox.
- Start the Mastra server explicitly after creating or resuming the sandbox.

### 6. Hono proxy routes (`app/api/[[...route]]/route.ts`)
- `/api/mastra/*` — proxy to the user's E2B sandbox (with auto-resume).
- `/api/llm/:provider/*` — virtual key resolution + proxy to real provider
  API. Must handle SSE streaming correctly.

### 7. Sandbox lifecycle (`lib/sandbox.ts`)
- `createSandboxForUser(userId)` — creates E2B sandbox, generates virtual
  key, stores everything in Supabase.
- `getSandboxForUser(userId)` — returns sandbox info from Supabase.
- `connectToSandbox(sandboxId)` — connects/resumes E2B sandbox.

### 8. Virtual key system (`lib/virtual-keys.ts`)
- `getOrCreateVirtualKey(userId)` — generates `vk_{userId}_{24hex}`, stores
  in Supabase `virtual_keys` table.
- `resolveVirtualKey(vk)` — looks up virtual key in Supabase, returns
  userId if valid.

### 9. Sandbox status endpoints
- `GET /api/sandbox/status` — reads status from Supabase, returns to frontend.
- `POST /api/sandbox/ready` — webhook called by sandbox, verifies token,
  updates status to `ready`.

### 10. Sandbox retry
- `POST /api/sandbox/retry` — resets sandbox status to `creating` and
  re-triggers `createSandboxForUser`.

### 11. `public/studio/` gitignore entry
Add to `.gitignore`:
```
public/studio/
```

### 12. Error handling
- `/api/mastra/*` returns `503` if sandbox is not created yet or fails to
  resume. Studio should handle this gracefully.
- The polling frontend should handle `{ status: 'error' }` with a retry
  button.

## Vercel Deployment Notes

- Build command: `npm run build` (runs `build:studio` + `next build`)
- Output directory: `.next` (default)
- Node.js version: 22.x
- The Hono catch-all route uses `export const runtime = 'nodejs'` — do not
  change to `'edge'` as `Sandbox.connect()` requires Node.js APIs.
- Vercel function max duration: set to 60s on Hobby, 300s on Pro. The
  `Sandbox.connect()` resume path is ~1s. The `createSandboxForUser()` path
  runs via `after()` so it does not count against the request timeout.
- `after()` requires Next.js 15 and Vercel — it keeps the serverless function
  alive after the response is sent.

## Security Notes

- `/api/sandbox/ready` is intentionally excluded from WorkOS middleware — it
  is called by the sandbox (no user session). It is protected by the one-time
  `READY_TOKEN` instead.
- `/api/llm/:provider/*` is not protected by WorkOS middleware — it is called
  by the sandbox. Protection comes from the virtual key: an invalid or unknown
  key returns `401`.
- The virtual key embeds `userId` in plaintext but is validated against
  the Supabase `virtual_keys` table — possession of a valid key proves
  identity without a session cookie.
- Real API keys never leave the Next.js server — they exist only in Vercel
  environment variables and are substituted in-memory during the proxy request.
- Authentication is handled through WorkOS Magic Auth with email OTP.
